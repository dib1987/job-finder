# Job Finder Agent — Developer Guide

## Architecture: WAT (Workflows, Agents, Tools)

```
Layer 1: Workflows  — workflows/job_application.md   (WHAT to do)
Layer 2: Agent      — agent.py                       (WHEN and sequencing)
Layer 3: Tools      — tools/                         (HOW — deterministic executors)
```

The agent orchestrates; tools execute. Tools never call each other. Tools never call the agent.

## Key Design Decisions

### 1. Job Source Abstraction
`tools/job_source/base.py` defines a `JobSource` ABC. `agent.py` only calls `source.fetch_jobs()`.
To swap from LinkedIn scraper to RapidAPI: set `JOB_SOURCE=rapidapi` in `.env`. Zero code changes.

### 2. LLM Abstraction
`agentic_base/utils/llm_client.py` provides `get_llm_client()` which reads `LLM_MODEL` from env.
Set `LLM_MODEL=gpt-4o` to switch to OpenAI. Same interface everywhere.

### 3. No Resume Tailoring (POC)
The agent sends the original `config/resume.pdf` as-is.
Resume tailoring (LLM modification) is a Phase 2 feature — deferred to avoid hallucination risk.

### 4. Human-in-the-loop (non-negotiable)
The approval gate (`tools/approval_tool.py`) is always required before any application is submitted.
`auto_apply_above` in preferences allows bypassing for very high scores (≥90) — opt-in, off by default.

### 5. Duplicate Prevention
`tools/tracker_tool.py` checks SQLite before each apply.
Running the agent daily will never re-apply to a job you've already applied to.

## Extending This Agent

### Add a new job source (e.g., Indeed)
1. Create `tools/job_source/indeed_scraper.py`
2. Implement `JobSource` ABC (fetch_jobs, get_job_detail)
3. Add to factory in `tools/job_source/base.py`:
   ```python
   if source_type == "indeed":
       from tools.job_source.indeed_scraper import IndeedScraper
       return IndeedScraper()
   ```
4. Set `JOB_SOURCE=indeed` in `.env`

### Add resume tailoring (Phase 2)
1. Create `tools/tailoring_tool.py`
2. Add `phase_tailor()` to `agent.py`
3. Insert "tailor" into `PHASES` list between "score" and "approve"
4. `apply_tool.py` already handles custom resume paths — pass tailored path instead of original

### Convert approval gate to web UI (SaaS)
1. Replace `_prompt_decision()` in `approval_tool.py` with a WebSocket event
2. Agent blocks waiting for the callback — same interface
3. No changes to agent.py or any other tool

## Troubleshooting

| Problem | Fix |
|---|---|
| `Session expired` in scraper | `python agent.py --setup-linkedin` |
| `Resume not found` | Copy resume to `config/resume.pdf` |
| `ANTHROPIC_API_KEY not set` | Add key to `.env` |
| Scoring returns 0 for all jobs | Check LLM response in logs — likely prompt format issue |
| Easy Apply button not found | Job may not support Easy Apply; marked as `manual_apply_needed` in tracker |
| LinkedIn blocking (CAPTCHA) | Reduce `MAX_JOBS_PER_RUN`, increase delays in linkedin_scraper.py |

## Shared Framework (agentic_base)

This agent inherits from `agentic_base.base_agent.BaseAgent`.
Other agents in this workspace (HospitalService, SocialMedia) can migrate to use the same base.

The shared interfaces are:
- `DataSource` — any "fetch items from external source" pattern
- `LLMScorer` — any "score items against criteria using LLM" pattern
- `StateManager` — `.tmp/` file-based state between phases
- `LLMClient` — provider-agnostic LLM calls
